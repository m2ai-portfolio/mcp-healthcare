

# MCP-Healthcare: Clinical Decision Support MCP Server with Compliance Guardrails - App Specification

## Overview
This specification defines a self‑contained Model Context Protocol (MCP) server that provides clinical decision support tools such as drug interaction checking, diagnostic criteria evaluation, and care pathway recommendations. The server runs entirely offline, enforces HIPAA‑compliant audit logging, and exposes its capabilities through a command‑line interface that can be invoked by existing Claude‑powered agents or other local services. By packaging the decision‑support logic as an MCP service, the solution addresses the gap in patient‑facing CDS systems where external APIs and cloud dependencies are prohibited.

**Problem Statement**  
Healthcare environments often require clinical decision support that must operate without external network calls, guaranteeing data locality and HIPAA compliance while integrating with agent‑based AI workflows.

**Target Audience**  
Clinical IT engineers, healthcare software developers, and compliance officers who need to embed reliable, auditable CDS functionality into local‑only AI applications.

## Tech Stack
- Python 3.11+ – core language
- Click 8.1+ – CLI command parsing and sub‑command handling
- Pydantic 2.6+ – data validation and settings management
- SQLite3 – embedded relational database for audit logs and reference data
- Pytest 8.0+ – test framework for unit and integration tests

## Environment Setup

### Prerequisites
- Python interpreter 3.11 or newer
- Access to a local filesystem with write permissions for the SQLite database
- Optional: a local copy of common drug interaction datasets (CSV/JSON) and diagnostic rule sets (JSON)

### Configuration
| Variable | Example | Description | Required |
|----------|---------|-------------|----------|
| `MCP_DATA_DIR` | `/var/mcp-healthcare` | Directory where SQLite file and static resources are stored | Yes |
| `MCP_DB_NAME` | `mcp_hc.sqlite` | SQLite database filename (relative to `MCP_DATA_DIR`) | Yes |
| `MCP_LOG_LEVEL` | `INFO` | Logging verbosity (DEBUG, INFO, WARNING, ERROR) | No (defaults to INFO) |
| `MCP_AUDIT_ENABLE` | `true` | Toggle HIPAA‑style audit logging | No (defaults to true) |
| `MCP_PORT` | `50051` | Local TCP port for MCP stdio communication (if used) | No (defaults to disabled) |

*All components execute in a single OS process; no threads, no external network calls.*

## Core Features
### Feature 1: Drug Interaction Checker
**Description**: Evaluates a patient’s medication list against a curated interaction database and returns severity‑graded alerts (contraindication, warning, precaution).  
**Requirements**:
- Accept a JSON array of medication objects (`{name, dose, route}`).
- Reference the latest interaction tables shipped inside `MCP_DATA_DIR/interactions.sqlite`.
- Output a JSON list of interaction alerts, each with `severity`, `description`, `source`.
- Write an audit record for every query (timestamp, user‑id, query‑hash, result‑count).
**Test Steps**:
1. `python -m pytest tests/test_drug_interaction.py::test_empty_list` → expects `[]`  
2. `python -m pytest tests/test_drug_interaction.py::test_known_interaction` → expects one alert with severity `"contraindication"`  
3. `python -m pytest tests/test_drug_interaction.py::test_audit_write` → verifies a new row in `audit_log` table  

### Feature 2: Diagnostic Criteria Evaluator
**Description**: Checks whether a set of clinical observations satisfies a selected diagnostic rule set (e.g., sepsis SIRS, COPD GOLD) and returns a boolean pass/fail with supporting evidence.  
**Requirements**:
- Receive a JSON observation bundle (`{vitals, labs, history}`).
- Load rule definitions from `MCP_DATA_DIR/rulesets/<name>.json`.
- Evaluate all required criteria; return `{passed: bool, evidence: [string]}`.
- Append an audit entry for each evaluation.
**Test Steps**:
1. `python -m pytest tests/test_diagnostic.py::test_met_criteria` → expects `passed: true`  
2. `python -m pytest tests/test_diagnostic.py::test_missing_labs` → expects `passed: false` with evidence listing missing lactate  
3. `python -m pytest tests/test_diagnostic.py::test_audit` → confirms audit row contains rule set name  

### Feature 3: Care Pathway Recommender
**Description**: Given a diagnosis and patient context, suggests an ordered set of care actions (medication, consult, monitoring) aligned with local guidelines.  
**Requirements**:
- Accept `{diagnosis: string, context: object}`.
- Pick the highest‑priority pathway from `MCP_DATA_DIR/pathways/<diagnosis>.json`.
- Output a JSON array of ordered steps, each with `action, detail, estimated_hours`.
- Log the recommendation event to the audit table.
**Test Steps**:
1. `python -m pytest tests/test_pathway.py::test_known_diagnosis` → returns non‑empty ordered list  
2. `python -m pytest tests/test_pathway.py::test_unsupported_diagnosis` → returns empty list and logs warning  
3. `python -m pytest tests/test_pathway.py::test_audit_integrity` → checks that audit entry includes recommendation count  

## Data Models
### MedicationInput (pydantic.BaseModel)
```python
class MedicationInput(BaseModel):
    name: str = Field(..., description="Drug name as reported by the prescriber")
    dose: str = Field(..., description="Quantity and unit, e.g., '500 mg'")
    route: str = Field(..., description="Administration route, e.g., 'PO', 'IV'")
```

### ObservationBundle (pydantic.BaseModel)
```python
class ObservationBundle(BaseModel):
    vitals: Dict[str, float] = Field(..., description="Blood pressure, heart rate, temperature, SpO2")
    labs: Dict[str, float] = Field(..., description="Selected laboratory results, e.g., creatinine, lactate")
    history: List[str] = Field(..., description="Relevant past diagnoses or procedures")
```

### InteractionAlert (pydantic.BaseModel)
```python
class InteractionAlert(BaseModel):
    severity: Literal["contraindication", "warning", "precaution"] = Field(...)
    description: str = Field(..., description="Human‑readable explanation of the interaction")
    source: str = Field(..., description="Reference identifier, e.g., 'Stockley 7e' or 'Micromedex'")
```

### PathwayStep (pydantic.BaseModel)
```python
class PathwayStep(BaseModel):
    action: str = Field(..., description="Clinical action, e.g., 'Administer antibiotic', 'Order CBC'")
    detail: str = Field(..., description="Free‑text specifics or dosage")
    estimated_hours: Optional[float] = Field(..., description="Rough time to complete the step")
```

## Success Criteria
1. **Functional Correctness** – After 5 iterations the server returns accurate interaction alerts, diagnostic evaluations, and pathway recommendations matching a curated test dataset (≥ 90 % concordance).  
2. **Audit Completeness** – Every CDS inquiry creates an immutable, timestamped audit record stored in SQLite that includes request hash, user‑id (≥ 5 iterations) and outcome summary.  
3. **Startup Latency** – The CLI command `mcp-healthcare --help` prints usage within 200 ms of process start on a modest VM (2 vCPU, 2 GB RAM).  
4. **Dependency Bound** – The runtime imports no more than **5** third‑party pip packages (click, pydantic, sqlite3, pytest, typer).  
5. **Offline Operability** – With all external network interfaces disabled, the server still processes requests and returns correct results using only local files and bundled reference data.

## Constraints & Notes
- **No External API Calls** – All data must reside in the `MCP_DATA_DIR` directory; the code must not import `requests`, `urllib`, `httpx`, or any cloud‑SDK client.  
- **HIPAA‑Style Audit** – Enable `MCP_AUDIT_ENABLE=true` to write an append‑only table `audit_log` with columns `(ts INTEGER PRIMARY KEY, user_id TEXT, query_hash TEXT, response_summary TEXT)`.  
- **Single Process Execution** – The design intentionally avoids `threading`, `multiprocessing`, `asyncio`, or any worker‑queue model to satisfy the “zero external dependencies” and “single data store” constraints.  
- **Local‑Only Distribution** – The reference datasets (interaction tables, rule sets, pathways) are shipped as read‑only files inside the package; they must not be downloaded at runtime.  
- **CLI‑Only Interface** – The server exposes capabilities solely through sub‑commands (`mcp-healthcare drug-check …`, `mcp-healthcare diagnose …`, `mcp-healthcare recommend …`); no HTTP, gRPC, or WebSocket listeners are instantiated.  
- **Keep It Small** – Aim for a compressed distribution (< 10 MB) that contains the SQLite DB (≈ 2 MB) and static JSON assets (≈ 5 MB); this assists environments with strict size limits.  
- **Test‑First Mentality** – All features are driven by the pytest test suite; new contributions must pass existing tests and add at least one corresponding test case.
